## 1. 프로젝트 개요

### 1.1 목적

이 프로젝트는 **UR 로봇 + AprilTag 비전 + Well Plate 좌표 변환**을 결합해, 운영자가 GUI에서 버튼을 누르면 다음 기능을 수행하도록 구성된 통합 제어 시스템입니다.

- Alvium 카메라 프레임 수집(실시간, Global Shutter)
- AprilTag(`tag36h11`) 검출 및 포즈 추정(pupil-apriltags)
- 태그 중심으로 “카메라/툴 장착각(기본 -60°)” 보정이 포함된 XY 상대 이동
- 4x6 Well Plate 각 웰(A1~D6)로 이동(태그 기준 로컬 좌표 → 카메라 좌표 → 로봇 상대 이동)
- 피펫팅 시퀀스(오프셋 이동 → Z 분할 하강 → Z 분할 상승 → 원위치)
- ROS 2 기반 UR 드라이버/MoveIt2 IK 연동 및 Trajectory 전송

### 1.2 전체 실행 구조(프로세스/모듈)

구조를 한 줄로 요약하면:

> **`main.py`(Tkinter GUI) ↔ `URInterface`(ROS2/MoveIt IK/Trajectory) + `TagDetectorWrapper`(카메라/AprilTag) + `AutomationController`(시퀀스/보정 로직)**

주요 흐름:

1. `main.py`가 Tkinter GUI(`CalibrationApp`)를 띄웁니다.
2. 같은 프로세스에서 ROS2 노드(`URInterface`)를 만들고, `SingleThreadedExecutor`를 별도 스레드로 spin 합니다.
3. 카메라(`AlviumCamera`)를 별도 스레드로 스트리밍하며, `TagDetectorWrapper`가 프레임을 가져와 AprilTag를 추정합니다.
4. GUI 버튼 동작은 `AutomationController`로 위임됩니다.
   - 태그 중앙 정렬(최대 N회 피드백)
   - Well 이동(태그 포즈를 기준으로 목표 웰 위치를 계산 후 상대 이동)
   - 피펫 시퀀스(Z 분할 이동 포함)

### 1.3 왜 이렇게 구성했는가(설계 의도)

- **ROS2/MoveIt과 GUI를 한 프로세스에서**: 운영자 피드백(영상/상태)과 로봇 이동 명령을 즉시 연결.
- **카메라/태그 처리와 로봇 제어를 느슨하게 결합**: `TagDetectorWrapper.latest_result`를 통해 최신 프레임/태그 포즈만 공유.
- **툴 장착각(-60°) 보정**: 카메라 좌표계에서 계산한 오차를 로봇/툴 좌표로 변환하는데 보정이 필요.
- **Z 분할 이동(충돌/정밀도)**: 하강을 다점 궤적으로 만들어 급격한 이동/충돌 위험을 줄임.

├── robot_interface.py
├── vision_module.py

### 2.1 각 파일 역할

####  `main.py`
- **GUI 진입점**. Tkinter UI 구성, 영상 표시, 버튼 이벤트 → 컨트롤러 호출.
####  `automation_controller.py`
- **자동화/시퀀스 로직 핵심**. 60도 보정 상대 이동, 태그 센터링, Well 이동, 피펫 시퀀스.
####  `robot_interface.py`
- **ROS2 로봇 인터페이스**. `/joint_states` 구독, TF 조회, MoveIt `/compute_ik` 서비스, FollowJointTrajectory 액션 전송.
####  `vision_module.py`
- **비전 모듈**. AprilTag 검출, 포즈 추정, 화면 오버레이/축 표시, 회전 오프셋 적용.
####  `alvium_driver.py`
- **Alvium 카메라 드라이버(vmbpy)**. 카메라 스트리밍 스레드 및 최신 프레임 제공.
####  `config.py`
- **상수/파라미터 집합**. 로봇 링크/그룹, 태그/플레이트 치수, 제어 파라미터(분할 이동, 센터링 임계 등), 피펫 오프셋.

## 3.2 각 코드 및 함수별 상세 설명

### 3.2.0 좌표계/부호/단위 요약(가장 중요)

이 섹션은 인수인계 시 **가장 먼저 합의해야 하는 “부호/축/단위 규칙”**을 고정합니다.  
현장에서 “로봇이 반대로 움직인다 / 축이 바뀐 것 같다” 이슈가 나면, 아래 표를 기준으로 원인을 빠르게 좁힙니다.

#### 단위

- **로봇/이동 입력 단위**: m (미터)
- **Well/플레이트 치수**: mm(설정) → m(계산 시 변환)
- **태그 포즈 `t_vec`**: m
- **화면 픽셀 오차**: px

#### 좌표 프레임(개념)

- **카메라 프레임**: `TagDetectorWrapper`가 추정한 태그 포즈(`R_mat`, `t_vec`)가 정의되는 좌표계
- **로봇 base 프레임**: `RobotConfig.BASE_LINK` (= `base_link`)
- **로봇 TCP/툴 프레임**: `RobotConfig.EE_LINK` (= `tool0`)
- **플레이트 로컬 프레임**: 태그를 기준으로 정의한 Well 좌표(로컬)

#### 핵심 부호/축 규칙(문제 발생 시 체크 순서)

1) **카메라 픽셀 오차 → 카메라 좌표 오차 근사**
- 센터링에서 픽셀 오차 \(u\_err, v\_err\)를 내참으로 환산해 \(x\_{cam}, y\_{cam}\)를 근사합니다.
- 깊이(태그까지 Z)가 크면 같은 픽셀 오차라도 \(x\_{cam}, y\_{cam}\)가 커집니다(과/소보정 원인).

2) **카메라 오차 → 로봇 상대 이동** (`AutomationController.execute_relative_move`)
- 입력 `dx_cam, dy_cam`은 **카메라 좌표계에서의 상대 이동량(m)**으로 취급합니다.
- 코드에서 **축 스왑**이 들어갑니다:
  - `dx_p, dy_p = dy_cam, dx_cam`
  - 이 규칙은 **카메라 설치 방향/좌표계 정의와 로봇 축의 대응**을 맞추기 위해 존재합니다.
- 이후 **장착각 보정**(`RobotConfig.MOUNT_ANGLE_DEG`, 기본 -60°)을 적용해 툴 좌표계 이동량으로 변환합니다.
- 마지막으로 현재 TF의 회전(`R_base_tool`)로 **툴 좌표 → base 좌표 상대 이동**으로 바꿔 목표 위치를 만듭니다.

3) **태그 로컬(플레이트) 좌표 → 카메라 좌표** (`AutomationController.move_to_well`)
- `(row, col)`로부터 로컬 좌표 \(P\_{local}\)를 만들고,
- `P_well_in_cam = R_mat @ P_local + t_vec`로 카메라 기준 목표점을 구합니다.
- 최종 이동 명령은 `execute_relative_move()`로 들어가므로, **로컬축 정의/부호가 잘못되면 웰 방향이 뒤집힙니다.**

운영 팁:
- “반대로 움직임”이 생기면 보통 아래 중 하나입니다.
  - `MOUNT_ANGLE_DEG`가 실제 장착각과 다름
  - 축 스왑/부호가 설치 방향과 불일치
  - 태그 회전 오프셋(`TAG_*_OFFSET_DEG`)이 잘못되어 웰 방향이 틀어짐

### 3.2.1 가장 중요한 실행 흐름(인수인계 추천 순서)

1. `main.py` (`CalibrationApp`): GUI 이벤트가 어디로 연결되는지 확인
2. `automation_controller.py` (`AutomationController`): 이동/시퀀스의 “의도”가 구현된 곳
3. `robot_interface.py` (`URInterface`): IK/Trajectory 전송 및 ROS2 의존 포인트 확인
4. `vision_module.py` (`TagDetectorWrapper`): 태그 포즈가 어떤 좌표/보정으로 만들어지는지 확인
5. `config.py`: 모든 숫자(오프셋/간격/임계)가 어디서 오며 무엇을 의미하는지 확인
6. `run.sh`: 실제 현장 실행 프로세스 확인

### 3.2.2 `AutomationController.execute_relative_move` (60도 보정 상대 이동)

입력 `dx_cam, dy_cam`은 **카메라 좌표계에서의 오차(m)**로 취급됩니다.  
이를 다음 순서로 변환합니다.

- 툴 장착각 `RobotConfig.MOUNT_ANGLE_DEG`(기본 -60°)를 적용해 **카메라 오차 → 툴 좌표계 이동량**으로 변환
- 현재 TF의 툴 회전 `R_base_tool`을 적용해 **툴 좌표계 → base_link 좌표계 상대 이동량**으로 변환
- 최종 목표 위치(`target_pos`)를 만들고 `URInterface.solve_ik_and_move()`로 IK/이동을 수행

주의:

- TF가 준비되기 전에는 `can_transform(..., timeout=2s)`로 대기 후 실패 시 반환합니다.

### 3.2.3 `AutomationController.center_tag_and_get_latest_pose` (태그 센터링)

- `TagDetectorWrapper.latest_result`에서 태그의 픽셀 중심(`center_px`)을 읽습니다.
- 이미지 중심과의 오차를 $u\_err, v\_err$로 계산합니다.
- 카메라 내참 `K`를 이용해 픽셀 오차를 카메라 좌표의 거리 오차로 근사합니다.
- `execute_relative_move()`로 오차를 줄이는 방향으로 반복 이동합니다.

운용 파라미터:

- 임계 픽셀: `ControlParams.CENTER_THRESH_PX`
- 최대 반복: `ControlParams.CENTER_MAX_ITERS`
- 반복당 타임아웃: `ControlParams.CENTER_PER_ITER_TIMEOUT`

### 3.2.4 `AutomationController.move_to_well` (Well 이동)

전제:

- **태그가 플레이트 기준 프레임**처럼 사용됩니다.
- `WellPlateConfig.OFFSET_TAG_TO_REF_Y_MM`와 `WellPlateConfig.WELL_SPACING_MM`로 로컬 좌표를 정의합니다.

계산 흐름:

1. 목표 웰(row, col)을 로컬 좌표 `P_local`(m)로 만든다.
2. 태그 포즈(`R_mat`, `t_vec`)를 이용해 카메라 기준 웰 위치를 구한다: `P_well_in_cam = R_mat @ P_local + t_vec`
3. `move_x, move_y`를 만들어 `execute_relative_move()`로 이동한다.
4. 이동 시간은 거리 기반으로 `ControlParams.WELL_MOVE_*` 범위에서 클램프된다.

### 3.2.5 `AutomationController.move_z_linear` (Z 분할/다점 궤적)

- 목표 Z 이동량 `dz_total`을 `ControlParams.LINEAR_STEP_M` 간격으로 나누고,
- 각 점에 대해 `URInterface.get_ik_sync()`로 IK를 미리 풀어 `q_sequence`를 만든 뒤,
- `URInterface.send_multi_point_traj()`로 하나의 다점 궤적을 보냅니다.

주의:

- IK 실패 시 즉시 False로 반환합니다(현장에선 안전상 중요).
- `get_ik_sync()`는 서비스 준비 여부/스레딩에 민감하므로, **GUI 스레드에서 블로킹 호출을 직접 늘리지 않는 방향**이 안전합니다.

### 3.2.6 `TagDetectorWrapper.process_latest_frame` (태그 추정)

- 카메라 프레임을 가져온 뒤 grayscale로 변환하여 AprilTag 검출
- `estimate_tag_pose=True`로 포즈(`pose_R`, `pose_t`)를 얻음
- 회전 오프셋(`WellPlateConfig.TAG_*_OFFSET_DEG`)을 적용한 `R_mat` 생성
- 디버그를 위해 화면에 축/텍스트 오버레이를 그린 `display_frame` 제공
- 결과는 `latest_result`에 저장되어 다른 계층이 읽음

### 3.2.7 `AutomationController.run_pipette_sequence` (피펫 시퀀스)

목표: 운영자가 “오프셋 → 하강 → 상승 → 복귀”를 한 번에 실행할 수 있도록 제공합니다.

흐름:

1. 현재 TF에서 시작 포즈를 읽습니다(`base_link`↔`tool0`).
2. **오프셋 이동(X/Y)**: 입력 `pip_offset_x`, `pip_offset_y`(m)를 적용해 XY로 이동합니다.
3. **하강(Z-)**: `move_z_linear(..., -pip_down_z, ...)`로 분할 하강합니다.
4. **상승(Z+)**: 바닥 위치에서 `move_z_linear(..., +pip_down_z, ...)`로 분할 상승합니다.
5. 시작 포즈로 복귀합니다.

주의(안전/부호):

- `pip_down_z`는 **양수(m)**로 입력되고, 실제 하강은 내부에서 `-pip_down_z`로 적용됩니다.
- Z 분할 이동은 스텝마다 IK를 풀기 때문에 **도달 불가/충돌 구간에서 중단될 수 있습니다.**
- 오프셋 값(`PipetteConfig.PIPETTE_OFFSET_*`)은 장착/툴 길이/작업물 위치에 따라 바뀌며, 실측 기반으로 관리하는 것을 권장합니다.

### 3.2.8 `URInterface.get_ik_sync` / `solve_ik_and_move` (IK 호출 패턴)

`URInterface`는 두 가지 IK 호출 패턴을 씁니다.

- `get_ik_sync()`:
  - **다점(Z 분할) 궤적 생성용**으로 사용합니다.
  - 블로킹 호출이므로 **GUI 메인 스레드에서 직접 호출을 늘리는 것은 피하는 것이 안전**합니다.
  - 전제: `/joint_states`를 최소 1회 이상 받아 `joint_order`와 `last_q`가 준비되어 있어야 합니다.
- `solve_ik_and_move()`:
  - 일반 이동용(비동기)입니다.
  - IK 완료 콜백에서 `send_traj()`로 Trajectory Action을 전송합니다.

실패 시 체크:

- `/compute_ik` 서비스 준비 여부, Action server 준비 여부, `BASE_LINK/EE_LINK` 프레임 이름 불일치 여부

### 3.2.9 `CalibrationApp._ui_tick` (UI/비전/로봇 상태 업데이트 루프)

`_ui_tick()`은 Tkinter 메인 스레드에서 주기적으로 호출되는 “UI 심장박동”입니다.

- 로봇:
  - `/joint_states`로 받은 관절각 표시
  - TF로부터 TCP 좌표 표시
- 비전:
  - `TagDetectorWrapper.process_latest_frame()`를 **매 tick마다 1회** 호출하여 최신 프레임 1장을 처리
  - 디버그 오버레이/조준선 표시 후 Tkinter 라벨에 투영
- 주기:
  - `root.after(40, ...)`로 약 25 FPS 수준(환경에 따라 변동)

주의(스레딩/성능):

- Tkinter 위젯 업데이트는 반드시 메인 스레드에서 해야 합니다.
- 프레임이 무겁거나 태그 검출이 느려지면 UI가 버벅일 수 있으므로, 이 경우 `Detector` 파라미터(예: `quad_decimate`)나 UI tick 주기를 조정합니다.

- **Tag family**: `tag36h11` (`WellPlateConfig.TAG_FAMILY`)
- **Tag size**: `WellPlateConfig.TAG_SIZE_M` (실측과 일치해야 포즈/거리 추정이 안정적)
- **부착 오차 보정**
  - 태그 부착 각도 오차는 `WellPlateConfig.TAG_*_OFFSET_DEG`로 보정(현장 튜닝 포인트)

## 4.3 운용 시 사용 방법(현장 체크리스트)

권장 운용 순서:

1. UR Driver/MoveIt/TF 정상 여부 확인(로봇 상태, TF tree, `/compute_ik` 준비)
2. `main.py` 실행 후 GUI에서 **카메라 영상 표시** 확인
3. 태그가 화면에 들어오면 **Tag 탐지 상태/좌표가 갱신되는지** 확인
4. `태그 중앙 정렬` 버튼으로 센터링(수 회 반복 이동)
5. Well 버튼(A1~D6)으로 목표 위치 이동
6. 필요 시 피펫 시퀀스(오프셋/하강/복귀) 실행

문제 발생 시 1차 확인:

- 태그가 탐지되는데도 로봇이 반대로 움직인다 → `RobotConfig.MOUNT_ANGLE_DEG`, 축 스왑(`execute_relative_move`) 확인
- 센터링이 수렴하지 않는다 → `ControlParams.CENTER_THRESH_PX`, 카메라 내참(`get_intrinsics`) 및 태그 크기(`TAG_SIZE_M`) 확인
- IK 실패가 잦다 → 로봇 초기 자세, `HOME_JOINTS`, MoveIt 파라미터/충돌, 목표 포즈 범위 확인
- Z 분할 이동 중 멈춘다 → `get_ik_sync`에서 특정 스텝 IK 실패, step 크기(`LINEAR_STEP_M`) 또는 목표 Z 확인

### 4.4 실패모드 트러블슈팅(증상 → 원인 → 조치)

| 증상(현장에서 보이는 것) | 가능 원인(자주) | 조치/확인 포인트 |
|---|---|---|
| GUI는 뜨는데 로봇이 전혀 안 움직임 | ROS2 executor spin 안 됨 / Action server 미준비 / `/joint_states` 미수신 | `main.py`에서 executor 스레드가 살아있는지, `URInterface.ac.server_is_ready()` 조건, `/joint_states` 들어오는지 확인 |
| “TF 기다림(Waiting for TF …)” 상태가 계속됨 | TF tree 미구성 / 프레임 이름 불일치 | `RobotConfig.BASE_LINK`, `RobotConfig.EE_LINK`가 실제 TF 프레임과 같은지, UR Driver/MoveIt 실행 여부 확인 |
| 센터링 버튼 눌러도 태그를 못 찾음 | 카메라 프레임 None / 조명/초점/태그 크기/가림 | 카메라가 정상 스트리밍인지(영상 표시), 태그가 화면에 충분히 크게 보이는지, `TAG_SIZE_M` 실측과 일치하는지 확인 |
| 센터링이 발산하거나 과하게 움직임 | 픽셀→m 근사에서 깊이(Z) 영향 / 내참 부정확 | 태그까지 거리 변화가 큰지, `get_intrinsics()`(초점/픽셀사이즈) 값이 맞는지, 임계/스텝을 완화(`CENTER_THRESH_PX`, move_time) |
| Well 버튼을 누르면 엉뚱한 방향으로 이동 | 로컬축 정의/부호 오류 / 태그 회전 오프셋 오류 | 3.2.0 좌표계 요약 기준으로 row/col 방향을 검증, `TAG_*_OFFSET_DEG`, `MOUNT_ANGLE_DEG` 재확인 |
| IK 실패가 뜨거나 Z 이동 중 특정 구간에서 멈춤 | 특정 포즈가 로봇 도달 불가 / 충돌/관절 한계 / seed 문제 | 시작 자세를 바꾸거나 목표 Z/step을 줄임, `LINEAR_STEP_M` 완화, MoveIt 충돌 설정 확인 |
| 앱 종료가 느리거나 멈춘 것처럼 보임 | 카메라 스트리밍 루프 종료 지연 / SDK timeout | `alvium_driver.py`의 스트리밍 timeout 특성(최대 수 초 지연 가능) 인지, 먼저 `detector.cam.stop()` 호출 후 종료 |

## 5. 인수인계를 위한 유의사항

### 5.1 가장 중요한 운영 전제

- **ROS2/MoveIt 의존**이 강합니다. `/compute_ik`, FollowJointTrajectory 액션, TF가 준비되지 않으면 대부분 기능이 동작하지 않습니다.
- **카메라 드라이버 의존**이 있습니다. Alvium을 쓰지 않는 환경에서는 `alvium_driver.py` ImportError로 대체 클래스가 사용되며, 이 경우 “카메라가 실제로는 동작하지 않을 수 있음”을 인지해야 합니다.
- 카메라 및 하드웨어 세팅으로 인한 반응속도 및 회전 오차가 발생할 수 있습니다.

### 5.2 코드 관점 주의 포인트

- `main.py`에서 ROS2 executor를 별도 스레드로 spin 합니다. 종료 시 `executor.shutdown()` 및 `rclpy.shutdown()` 순서/예외 처리가 중요합니다.
- `URInterface.get_ik_sync()`는 서비스 준비/스레딩에 민감합니다(블로킹 호출). 다점 궤적 생성에만 사용되고, 일반 이동은 비동기(`call_async`)를 씁니다.
- `TagDetectorWrapper.latest_result`는 “마지막 프레임 기준”입니다. 타임스탬프 기반 `min_timestamp` 가드가 센터링 품질에 중요합니다.

### 5.3 인수인계 추천 전달 순서(요약)

1. 터미널 2개(UR Driver/MoveIt) + 앱 1개로 전체 시스템 기동 방법 설명
2. `main.py`에서 GUI 버튼이 무엇을 호출하는지 설명
3. `automation_controller.py`에서 보정/시퀀스/Well 변환 로직 설명
4. `robot_interface.py`에서 ROS2/MoveIt 연동 포인트(토픽/액션/서비스/TF) 설명
5. `vision_module.py`에서 태그 추정/오프셋/내참 파라미터 설명
6. `config.py`의 상수들을 “현장 튜닝 포인트”로 정리해서 전달
